Recon first-pass plan

Scraper, backend, and database plan for monitoring Indonesian second-hand computer, tech, and gaming gear listings from public source URLs. This is not the UI plan.

  • Project: C:\Project\recons
  • Internal name: RECON
  • Public placeholder: Recon.co.id
  • Created: 2026-07-01
  • Updated: 2026-07-04
  • Status: first-pass scraper plan active

1. Core decisions

  • Product shape: scraper-first listing intelligence system, not a landing page and not a marketplace checkout product.
  • Initial market: Indonesia only. Listings may be Indonesian, English, or mixed Indonesian-English.
  • Initial categories: computer, PC parts, tech, gaming gear, and gaming peripherals.
  • Primary workflow: fetch public listings, normalize messy marketplace/social-post data, store provenance, and expose reliable queryable records.
  • Auth: no login, user accounts, saved preferences, or authenticated scraping in the first pass.
  • AI: optional AI field extraction is allowed for messy source text, but no recommendation, deal scoring, or market-price baseline in v1.
  • Scraping: public URL fetch only. No login credentials. No Playwright/Selenium unless explicitly approved as a last-resort fallback.
  • Scraper runtime: Python is locked for v1.
  • First connector: harden Reddit first because it is the easier public-fetch baseline.
  • Cadence: validate 5-minute checks in staging first. Production starts conservative and adapts per connector health.
  • Images: store original image URLs and cache small thumbnails for future gallery/hover interactions.
  • Sold listings: keep sold items permanently with explicit status and provenance.
  • Deployment: Dockerized scraper, backend, and PostgreSQL stack on homelab first, cloud-portable later.

2. First-pass boundary

Goals

  • Prove public source fetching and listing normalization can stay stable enough for daily use.
  • Store consistent listing rows from messy post descriptions.
  • Keep scraper run logs, connector health, failures, dedupe decisions, and ingestion counts in scraper-side logs or files.
  • Expose a backend query contract after real normalized listings exist.
  • Validate safe connector cadence before chasing near minute-level freshness.

Non-goals

  • No checkout, escrow, chat, negotiation, seller dashboard, or consignment workflow.
  • No alerts, watchlists, saved searches, user accounts, or personalization.
  • No deal score, price intelligence, price baseline, AI ranking, or recommendations.
  • No public UI implementation in the first-pass plan.
  • No cross-platform duplicate merge logic. Same product on different platforms can appear separately.

Success metrics

  • Freshness: in staging, each active connector can run every 5 minutes without duplicate spam, crashes, or source lockout.
  • Parse quality: 80%+ of stored listings have title, source URL, platform, availability state, and price/category/brand when confidently parsed.
  • Provenance: raw listing descriptions stay stored; parser evidence and confidence remain scraper-side until the database plan explicitly adds provenance columns.
  • Health: scraper-side logs show last success, last failure, and error reason.
  • Scope: no login, public UI, alerts, AI, payment, or chat in the first-pass release.

  • Gamers: look for gaming gear, keyboards, mice, headsets, and consoles.
  • IT students and hobbyists: track affordable used laptops, components, and accessories.

Scraping boundary

  • Use public URL fetch only: no login, no credential use, no bypassing access controls.
  • If fetch-only breaks, patch the connector first.
  • Browser automation can be proposed only as a separate decision after repeated connector repair fails.
  • Validate 5-minute checks in test/staging; production cadence is connector-specific.
  • VPN/proxy egress is not the first response to blocks. Slow down, back off, inspect errors, then mark degraded if needed.

4. Architecture

Public source URLs
  |-- Reddit connector (first hardened connector)
  |-- Instagram connector
  |-- Facebook Marketplace connector
          |
          v
Scraper service
  main.py -> selected source modules -> fetch -> parse -> normalize -> store
          |
          v
PostgreSQL
  listings, listing_images
          |
          v
Backend API / query layer
  listing feed/search contract -> future UI

Backend API boundary

  • API implementation can use Next.js + TypeScript when the web app scaffold starts.
  • tRPC or typed route handlers are acceptable if they keep contracts explicit.
  • Prisma remains the preferred database access layer.
  • Expose listing queries only after normalized database rows exist.
  • Keep connector health and scrape-run history outside the database unless the product later needs it.
  • Public UI implementation is deferred behind scraper, backend, and database readiness.

Scraper service

  • Separate runtime from the web app.
  • Locked v1 stack: Python.
  • Platform modules stay separate: reddit, instagram, and facebook.
  • One orchestrator runs all sources or selected sources, for example python main.py --reddit.
  • Records each scrape run with status, duration, source count, listing count, and error details.

Runtime shape

  • main.py: CLI entrypoint, flag parsing, source selection, run lifecycle.
  • sources/*: connector-specific fetch and parse logic.
  • normalizers/*: shared mapping from source payloads to listing contract.
  • storage/*: database upsert and thumbnail cache.
  • fixtures/*: saved public samples for parser regression tests.

Container stack

  • scraper: Python orchestrator, source modules, retry/backoff policy, normalized listing writes.
  • backend: future API/query layer.
  • postgres: separate staging and production databases.
  • storage: cached thumbnail storage, local mounted volume first, object-storage-compatible interface later.
  • vpn-egress: optional network route, enabled only after rate/backoff/degraded handling is reviewed.

5. Data model draft

Listing fields

  • id
  • platform: reddit, instagram, facebook
  • source_url: original post/listing URL
  • external_id: nullable source post/listing ID when available
  • title
  • description
  • category
  • brand
  • price: nullable integer IDR value when confidently parsed; no currency column in v1.
  • location_texts: one or more public listing locations.
  • condition_text
  • seller_name
  • status: available, sold, unknown
  • posted_at, first_fetched_at, last_fetched_at

Supporting records

  • listing_images: listing ID, source URL, position, and optional alt text.

Deferred price issues

  • Currency is assumed to be IDR in v1. If a source says USD or another currency while the listing is effectively Indonesian rupiah context, do not add a currency field yet; handle that later in parser rules.
  • Some sellers write shorthand prices, such as 1 for 1,000,000 or 100 for 100,000. Do not solve this in the first schema. Store price only when the parser is confident; otherwise leave it null and keep the original wording in description.

Deduplication

MVP dedupes only within the same platform by stable source URL or source post ID. It does not merge similar products across platforms.

Availability parsing

  • Available: #ready, #chemicyready, "ready", "available", or no sold marker. Store as available.
  • Unknown: caption cannot confidently identify available or sold. Store as unknown.
  • Sold: "sold", "sold out", "terjual", "laku", or "booked" when treated as unavailable. Store as sold.

6. Risks

  • Rate limits and blocks: use scraper-side health logs, backoff, degraded-source state, and explicit approval before browser fallback.
  • VPN overuse: require error inspection before egress changes; prefer slowing or pausing connectors over rotation.
  • Noisy parsing: keep original listing wording in description and use fixture-backed parser tests.
  • Thumbnail storage risk: cache thumbnails only, preserve source attribution, delete cached image on takedown, avoid full-size archive in v1.
  • Legal/platform risk: run focused review on platform terms, takedown handling, data retention, cached media, and Indonesian privacy obligations before public launch.
  • Homelab limits: start on homelab, keep Dockerized architecture cloud-portable, and monitor storage growth.

8. Release and deployment

Local development

  • Start with database and scraper development before any public UI.
  • Run PostgreSQL through Docker Compose.
  • Run scraper manually first with python main.py and source-specific flags.
  • Validate 5-minute scheduled checks in staging before using that cadence in production.
  • Use saved fixtures to test parser changes without hitting platforms repeatedly.

Homelab CI/CD

  • GitHub Actions builds and tests.
  • Docker images published under the user's Docker Hub namespace.
  • Staging deploys to the Debian VM first.
  • Scraper container can use normal direct egress or explicit VPN/proxy egress.
  • Production promotion should be manual and version-tagged.
  • Rollback should redeploy the previous fixed image tag.

Public launch checklist

  • Brand name and domain decision checked.
  • Reddit connector stable enough for daily feed freshness as the first hardened source.
  • Instagram connector stable or queued immediately after Reddit.
  • Facebook connector stable, marked beta, or deferred.
  • Five-minute staging checks tested with cooldown/backoff and no duplicate pileup.
  • Takedown/hide path implemented.
  • Platform-term risk reviewed and accepted or mitigated.
  • Privacy note: Recon does not collect visitor accounts in v1; listing data comes from public source posts.
  • Source URL and raw listing description preserved in stored rows and API responses; extraction confidence remains scraper-side until explicitly added to the schema.
